# Cortex 协作工作台需求稿

最近更新：2026-05-08

## 1. 背景与机会

- 当前问题：
  Cortex 已经有执行内核、红黄绿灯决策、Memory 治理和文档异步协作的雏形，但前端协作载体仍然分散。现在想看全局状态，要去 `/dashboard`；想看执行上下文，要进文档；想看评论是否触发了任务，还要理解底层对象。这导致系统能跑，但不够像一个稳定、连续、可感知的人机协作空间。
- 为什么现在做：
  我们已经确认 Notion / Obsidian 更适合作为镜像和补充入口，不适合作为 Cortex 的主协作界面。现在是把“已有后端能力”产品化成一个统一工作台的最佳时点。
- 不做的代价：
  后端会越来越强，但用户仍然只能靠零散页面和隐含规则理解系统；评论能不能触发执行、哪些任务卡住、哪些决策需要拍板，依旧不够直观，最终会拖慢 Cortex 从个人协作系统走向小团队协作系统。

## 2. 目标、成功指标、非目标

- 目标用户：
  当前以你自己为主，下一阶段面向 `1-10` 人以内的人机协作团队。
- 核心目标：
  交付一个统一的 Cortex 协作工作台，把“全局任务分诊”和“单文档异步协作”放到同一个产品框架里。
- 成功指标：
  1. 人可以在一个入口内看到红黄绿灯事项、线程状态和最近完成。
  2. 人可以在文档中直接评论并驱动下一步执行。
  3. Agent 执行后的 receipt / checkpoint 能回写到对应线程。
  4. 高价值裁定动作可以进入 Memory Extract 流程。
- 观察周期：
  先看 `1-2` 周真实自用，再决定是否扩到更多项目。
- 非目标：
  1. 不做 Notion 替代品。
  2. 不做完整多人实时协同编辑。
  3. 不做开放式聊天应用。
  4. 不做重型权限系统和企业管理后台。

## 3. 设计原则

### 3.1 OpenAgents 视角：工作台首先是协作空间

参考 OpenAgents，我们借的不是它的托管产品，而是它对 workspace 的理解：

- 一个持续存在的协作空间
- thread 是协作基本单位
- 文件、文档、线程、浏览上下文是共享对象
- 人和 agent 在同一个空间里协作，而不是各自待在独立聊天窗口里

翻译成 Cortex：

- `任务中台总览页` 负责空间总览
- `文档三栏页` 负责空间内的具体协作
- `thread / decision / receipt / memory` 是空间内的核心对象

### 3.2 gstack 视角：规划先拆角色，再拆阶段

当前环境里没有可直接调用的 `gstack` 本地 skill，所以这里借的是它的方法论：

- 先做产品/CEO 视角判断
- 再做设计视角判断
- 再做工程视角锁边界
- 再做开发体验和可观测性检查

翻译成 Cortex 需求规划：

- 产品层：这个工作台到底解决什么协作问题
- 设计层：用户最自然的导航、分栏和状态表达是什么
- 工程层：哪些能力复用，哪些要自己做
- DevEx 层：如何保证后续 agent 和工程协作成本不失控

### 3.3 Cortex 自身原则

- 本地 `Markdown + SQLite` 仍然是单一真相源
- 红黄绿灯决策机制不能被弱化
- Memory 是治理对象，不是聊天摘要
- 人不做调度器，只做关键决策者
- 文档评论必须是执行入口，而不是普通留言区

## 4. 产品定义

`Cortex 协作工作台 = 一个把全局任务分诊、单任务执行跟踪、文档异步协作、评论驱动执行、Memory 治理回流整合到一起的本地优先 workspace。`

它要解决两类问题：

1. `全局看不清`
   现在有哪些红灯、黄灯、绿灯事项，哪些在跑，哪些卡住，哪些最近完成。
2. `局部协作不连续`
   在具体文档里评论之后，系统是否理解了评论、是否转成任务、是否有人处理、处理结果是什么。

## 5. 人类判断与智能体执行边界

- 必须人工判断的事项：
  1. 红灯决策。
  2. `allow_as_default` 这类会改变长期协作规则的裁定。
  3. 需要改变产品方向或审美方向的决策。
- 可由智能体自动执行的事项：
  1. 绿灯任务直接推进。
  2. 黄灯先绕行并记录。
  3. 评论意图识别和任务入队。
  4. checkpoint / receipt 回写。
  5. Memory candidate 初筛。
- 需要升级人工的情形：
  1. 不可逆变更。
  2. 高风险或跨模块污染。
  3. 评论语义不清且不能安全猜测。
  4. thread 裁定可能改变默认协作规则。
- 人工覆盖/兜底方式：
  1. 在总览页直接看红灯待拍板和黄灯待回看事项。
  2. 进入具体文档 thread 做裁定。
  3. 红灯由系统通知或明确待拍板区升级。

## 6. 核心对象模型

P0 工作台围绕 7 类对象展开：

- `task`
- `document`
- `thread`
- `comment`
- `command`
- `decision`
- `receipt`

Memory 不是第一屏主对象，但它会从这些对象里被投影出来。

关键关系：

- 一个 `project` 包含多个 `task`
- 一个 `task` 是总览页中的一张卡片 / 一行记录
- 一个 `task` 归属于一个稳定的 `thread_key`，同时拥有一个给人看的 `thread_label`
- 一个 `task` 可以关联一个主文档，也可以关联多个 thread
- 一个 `task` 可以由不同 agent 在不同阶段接力执行
- 一个 `document` 包含多个 `thread`
- 一个 `thread` 包含多个 `comment`
- 一个 `thread` 可以关联 `task / command / decision / checkpoint / receipt`
- 一个 `thread` 的高价值裁定可以生成 `memory candidate`

## 7. 信息架构

## 7.1 双层视图

工作台不做成一个页面塞所有东西，而是两层：

### A. 任务中台总览页

用途：

- 看全局风险、堆积和推进状态
- 先决定该处理哪个任务
- 尽量替代你去多平台轮询

### B. 单文档三栏页

用途：

- 进入具体文档做编辑、评论、裁定和执行协作

这两层的关系是：

`总览页负责分诊，文档页负责执行。`

## 7.2 为什么默认看板比表格更适合 P0

你现在最痛的不是“查不到字段”，而是“任务停在某处你没及时看到”。

所以从用户视角，P0 默认入口更适合是 `看板`，而不是传统表格。

原因：

- 红灯任务会自然聚集在最显眼的一列
- 黄灯任务能形成次优先级队列
- 绿灯任务可以弱化展示，主要用于执行留痕和后续审计 review
- 你可以一眼看见“哪些任务需要我现在介入”
- 顶层组织逻辑更贴近“注意力分配”，而不是“字段管理”

这里的关键不是只做一种看法，而是：

- 同一首页
- 同一批 task
- 提供两种视图

这两种视图分别解决不同问题：

- `按注意力`
  - 帮你先分诊，知道哪些任务要立刻看
- `按线程`
  - 帮你理解某个协作线程下面一共拆了多少任务、哪些在跑、哪个红灯卡住了

表格不应该消失，但更适合做：

- 二级视图
- 筛选视图
- 审计视图

## 7.3 交互层合并，数据层不合并

从用户视角，`任务状态` 和 `决策状态` 可以部分合并。

因为你真正关心的是：

- 这个任务要不要我管
- 如果要管，是不是现在就要管

但从系统建模上，不能彻底合并。

因为下面这些状态在业务上并不等价：

- `执行中 + 红灯`
- `执行中 + 黄灯`
- `执行中 + 绿灯`
- `已完成 + 绿灯`

所以 Cortex 的正确做法是：

- `交互层`：默认按“关注优先级”合并成看板
- `数据层`：仍保留 `task_status` 和 `decision_signal` 两套字段

换句话说：

- 首页给人看的，是一个 `attention board`
- 系统底层保留的，仍然是 `执行进度` 和 `决策信号`

这样既符合你的用户心智，也不会把后续自动化判断做坏。

## 7.4 单任务卡片是 P0 的最小操作单元

你已经把最关键的一点说清楚了：

`一张卡片 = 一个具体任务`

而不是：

- 一个项目线程
- 一个 agent 执行流

一个项目里可以拆出多个任务。

每个任务：

- 可以挂一个主文档
- 可以关联多个评论 thread
- 可以由不同 agent 分阶段接力
- 可以有自己的红黄绿灯状态和执行状态

所以在顶层看板上：

- 一列不是一个项目阶段
- 一行也不是一个对话线程
- 一张卡片只表达“这个具体任务现在处于什么关注状态”

但首页还应该支持第二种组织方式：

- 同样是一批 `task`
- 不再按红黄绿分列
- 而是按 `thread_label` 分组

这样你能从另一个视角看到：

- 这个线程下面一共有几个任务
- 哪几个任务在跑
- 哪个任务已经红灯卡住

## 7.4.1 三层主单位不要混在一起

这里其实有 3 个不同层级：

- `task`
  - 用户真正要管理的工作单元，也是首页卡片单位
- `thread / decision point`
  - 任务内部某一个具体判断点，可能是 green / yellow / red
- `run`
  - 某个 agent 对某条 command 的一次实际执行过程

它们的关系是：

- 一个 `task` 可以有多个 `thread / decision point`
- 一个 `task` 也可以有多次 `run`
- `thread` 解决“这里要不要改路”
- `run` 解决“agent 现在到底跑到哪了”

所以首页不能只看 `run`，也不能只看 `thread`。

正确方式是：

- `task` 做顶层管理
- `thread` 解释为什么会变红、变黄、或继续绿灯
- `run` 提供执行脉搏，专门解决“agent 有没有悄悄卡住”

## 7.5 单文档三栏结构

- 左侧：文档目录
- 中间：编辑器
- 右侧：评论线程

右侧不是普通 comment rail，而是：

`评论线程 + 执行状态 + 人类裁定 + receipt/checkpoint`

## 8. 用户场景与主流程

### 场景 A：我想先看现在系统里哪些任务最需要我处理

- 触发条件：
  打开 Cortex 工作台首页。
- 输入：
  当前 project 的 task / command / decision / thread / receipt 聚合数据。
- 系统处理：
  聚合任务关注状态、执行状态和最近完成。
- 输出：
  一个默认按关注优先级组织的任务看板。
- 失败处理：
  如果 task 对象还没完全前端化，就先展示“基于 decision / inbox / command 的任务代理视图”。

### 场景 B：我进入某份文档，给出评论，希望系统继续推进

- 触发条件：
  人在文档里划词评论或回复 thread。
- 输入：
  选区、评论正文、当前 thread 上下文。
- 系统处理：
  comment intent classifier 判断这条评论是继续、修改、重做、提问还是拒绝。
- 输出：
  线程状态更新；如果可执行，则创建 command 并等待 agent claim。
- 失败处理：
  如果意图不清，则进入 triage，而不是误执行。

### 场景 C：我想把某次裁定直接变成长期协作默认规则

- 触发条件：
  在 thread 上选择 `允许，以后无需询问`。
- 输入：
  当前 thread 的文本、关联对象和人类裁定。
- 系统处理：
  允许本次继续执行，同时生成 memory candidate。
- 输出：
  后续进入 reviewer / curator 流程，决定是否升级为 durable memory。
- 失败处理：
  证据不足时只保留为 candidate，不自动升 durable。

## 9. 本期范围与下一阶段范围

- 本期范围：
  1. 双层视图：总览页 + 三栏文档页。
  2. 文档目录、编辑器、评论线程三栏布局。
  3. 评论到任务执行的最小闭环。
  4. 红黄绿灯和线程状态的可视化。
  5. 高价值裁定动作接入 memory candidate。
- 下一阶段范围：
  1. 多项目切换。
  2. 更强的 thread 筛选和批处理。
  3. 更自然语言的评论理解。
  4. 更完整的 memory review center。
  5. 小团队多角色协同。
- 明确不做：
  1. 实时多人协作编辑。
  2. 聊天式 agent 工作区。
  3. 外部 SaaS 文档平台替代。

## 10. 关键规则、不变量与约束

- 权限与角色：
  人类拥有最终裁定权；agent 只能在 guardrail 内自动推进。
- 数据校验：
  thread、comment、command、receipt 必须有稳定 ID 和关联关系。
- 隐私/合规：
  本地真相源优先，外部文档平台只做镜像或入口。
- 延迟/性能：
  P0 先保证本地顺畅，不追求大规模实时协作。
- 成本上限：
  不引入重型商业评论/协同基础设施作为前置依赖。
- 可靠性/SLA：
  评论不能因意图识别失败而直接误执行；默认宁可进入 triage。
- 禁止行为：
  1. 没有证据的 thread 裁定直接升 durable memory。
  2. 评论语义不明时强行执行。
  3. 把红灯伪装成黄灯或绿灯。

## 11. 线程状态设计

线程状态拆成两层：

### 11.1 执行状态

- `pending`
- `triaged`
- `ready`
- `running`
- `waiting_human`
- `done`
- `rejected`

### 11.2 人类裁定结果

- `allow_once`
- `allow_as_default`
- `reject_with_guidance`

对应展示文案：

- `允许`
- `允许，以后无需询问`
- `拒绝，并给出指导意见`

这是 Cortex 和普通评论系统最不一样的地方。

## 12. 总览页需求

总览页不是第二套系统，而是现有 `/dashboard` 的正式产品化。

它应该至少回答：

1. 现在一共有多少红灯、黄灯、绿灯任务。
2. 哪些任务在线等人。
3. 哪些任务正在执行。
4. 哪些任务刚完成。

P0 默认视图建议是 `看板`，不是表格。

但这里的“看板”不只是一种组织方式，而是同一首页的两种视图：

### 视图 A：按注意力

这是默认视图。

它回答：

- 哪些任务正在等我拍板
- 哪些任务还在系统处理中
- 哪些任务已经完成

### 视图 B：按线程

这是同一批 task 的分组视图。

它回答：

- 某个 `thread_label` 下面一共有多少任务
- 哪几个任务在跑
- 哪个任务是红灯
- 这个线程最近有没有推进

默认列建议：

- `红灯待拍板`
- `黄灯绕行中`
- `自动推进中`
- `已完成`

说明：

- `红灯待拍板`
  - 用户现在最需要关注
- `黄灯绕行中`
  - 当前有不确定性，但系统会先继续做其他 green 工作
  - 等同任务内 green 路径跑完后，再自动回看这个点
- `自动推进中`
  - 绿灯为主，agent 正在自行推进
- `已完成`
  - 主要用于回看、审计、复盘

这里的列含义是“关注优先级”，不是“生命周期阶段”。

也就是说：

- 用户先看 `红灯待拍板`
- 再回看 `黄灯绕行中`
- `自动推进中` 默认只需感知，不需要频繁盯着
- `已完成` 主要承担审计、复盘和结果确认

这里要特别注意：

- `yellow` 不是“现在就停下来等你处理”
- `yellow` 是“这里先记下来，先绕过去，把能安全做完的 green 先做完”
- 后续它有 3 种去向：
  - 随着上下文补全，降回 `green`
  - 继续保留为 `yellow`，等待异步评论或回看
  - 当它变成关键路径阻塞点时，升级为 `red`

P0 看板卡片内最小字段：

- 任务标题
- 主责 agent
- 当前执行状态
- 当前决策信号
- 线程标签 `thread_label`
- 优先级
- 最近更新时间
- 主文档入口

P0 线程分组视图的线程头最小字段：

- `thread_label`
- `thread_key`
- 总任务数
- 运行中任务数
- 红灯任务数
- 最近更新时间

P0 仍保留一个 `表格视图` 作为二级视图。

它适合做：

- 批量筛选
- 多字段排序
- 审计追踪

P0 总览页最小列表区：

- 需要立即拍板
- 黄灯绕行与待回看
- 最近完成

## 13. 文档三栏页需求

### 左栏

- 展示项目文档目录
- 显示当前文档
- 标识文档未处理 thread 数量

### 中栏

- 支持基础文档编辑
- 支持 block 结构
- 支持划词创建评论
- 支持与本地 Markdown / JSON 同步

### 右栏

- 展示 thread 列表
- 展示 thread 当前状态
- 展示 classifier 判断结果
- 展示裁定动作
- 展示关联 receipt / checkpoint
- 支持回跳正文锚点

## 14. 技术策略

- 编辑器内核：
  优先 `Plate`
- 评论锚点：
  复用 editor comment/annotation 能力
- 线程语义层：
  Cortex 自己做
- 评论意图识别：
  复用现有 `src/comment-intent.js`
- 真相源：
  继续使用本地 Markdown + SQLite

选择这条路的原因：

- 借到成熟编辑能力
- 保住 Cortex 语义层
- 不被商业协作能力绑死
- 不推翻现有执行后端

## 16. 埋点、日志、指标与评测

- 关键埋点：
  1. thread_created
  2. comment_intent_classified
  3. thread_resolution_selected
  4. command_enqueued_from_thread
  5. receipt_attached_to_thread
  6. memory_candidate_projected_from_thread
- 关键日志字段：
  project_id、document_id、thread_id、comment_id、intent、execution_policy、resolution、owner_agent、command_id
- 核心业务指标：
  1. 评论到执行转化率
  2. thread 从创建到完成的平均时长
  3. 红灯事项积压数
  4. 黄灯绕行后重新收口率
- 过程质量指标：
  1. 评论误执行率
  2. triage 命中率
  3. receipt 回写成功率
  4. memory candidate 命中率
- AI 评测样本与判定标准：
  用真实中文评论样本验证 continue / improve / retry / question / feedback / reject 分类。
- 人工抽检方式：
  每轮迭代抽查高风险 thread 的分类和执行结果。
- 告警条件：
  1. 红灯事项持续堆积。
  2. classifier 低置信度评论大量增长。
  3. receipt 无法回写 thread。

## 17. 验收标准

- 功能验收：
  1. 能打开总览页看到核心计数。
  2. 能进入某份文档进行编辑和评论。
  3. 评论能被系统分类并正确显示状态。
  4. 可执行评论能进入命令队列。
  5. receipt / checkpoint 能回写到 thread。
- 质量验收：
  1. 评论状态变化可追踪。
  2. 线程和命令关联清晰。
  3. 总览页和文档页信息一致。
- 异常场景验收：
  1. 语义不明评论进入 triage。
  2. 危险指令被拒绝。
  3. receipt 回写失败时 thread 不误判 done。
- 数据与观测验收：
  1. 关键埋点和日志字段齐全。
  2. 可以回放 thread 到 command 的链路。

## 18. 发布、灰度、回滚

- MVP 范围：
  单项目、单人主用的 Cortex 协作工作台。
- 试点对象：
  `PRJ-cortex`
- feature flag：
  以独立 route 或独立页面方式灰度，不替换现有所有页面。
- 灰度计划：
  1. 先把总览页正式化。
  2. 再做文档三栏页。
  3. 最后接通评论到执行闭环。
- 回滚条件：
  若 thread -> command 路径误执行率过高，则退回只读线程，不自动入队。
- 降级方案：
  保留现有 `/dashboard` + 本地 Markdown 文档视图。

## 19. 风险、待确认项、技术债

- 主要风险：
  1. 评论锚点和 Markdown 真相源之间的同步设计比想象中更复杂。
  2. thread 状态和现有 command / decision 对象之间可能会有双写风险。
  3. 评论语义分类的边界还需要更多真实样本。
- 待确认项：
  1. thread 是否先存 block anchor，再逐步支持文本 anchor。
  2. 总览页里“绿灯事项”是按 thread、decision 还是统一事项对象统计。
  3. 文档树是否只展示本项目文档，还是兼容 memory / decision 专用文档。
- 已接受的取舍：
  1. 先不做实时多人协作。
  2. 先不做复杂权限。
  3. 先不做完全自由对话式 agent 界面。
- 技术债与后续优化：
  1. memory review center 还未前端化。
  2. thread 与 memory 的闭环还会继续增强。
  3. 后续可能需要把 thread event stream 做成统一对象层。
