# 协作方式与工程审美

用于多 agent 共享的稳定协作记忆。

本地 markdown 是一等真相源。
Notion memory page 是共享镜像，不单独发散。

## Cortex Memory 定位

- 这份文档在 Cortex Memory Extract 结构里，默认属于 `durable + base_memory`。
- 主要承载的 type 是：
  - `preference`
  - `rule`
  - `pattern`
  - `decision`
- 默认挂载方式：
  - 新的 Cortex 协作任务启动时，作为 collaboration base memory 优先挂载
- 默认更新规则：
  - 用户明确说出的稳定偏好或长期规则，可以直接同步进 durable memory
  - 重复出现、但还带推断成分的模式，先按 candidate 判断，再决定是否晋升
  - 一次性情绪、临时 workaround、狭窄项目细节，不默认进入这份长期库

## 协作方式偏好

- 少确认。低风险事项直接推进，不等许可。
- 先做后验收。先产出结果，再一起 review。
- Agent 先往前推进，洛思再一起看结果。
- agent 要承担端到端推进责任，不把人当中转站。
- 默认持续推进。只有真正红灯才停下来等拍板。
- 低风险且明显有益的下一步，默认直接执行，不先问“要不要继续”。
- 如果下一步只带来收益、没有新增风险，默认判定为绿灯并直接执行，不返回“如果你要我可以继续”这种等待式收尾。
- 先执行，再汇报。想清楚下一步后直接调用工具执行，不只口头描述“接下来会做什么”。
- 回复里要有对应 action 和结果证据。没有实际 action 的回复，视为没有执行，不算自主推进。
- 不留小尾巴。发现残留问题时默认当场清掉，不把明显收尾工作留到下一轮。
- 明显冗余的旧页面、旧数据库、旧入口，默认直接归档，不只改名字占位。
- 默认循环是：实现 -> 验证 -> 总结 -> 一起验收。

## 任务简报契约

每次新任务先显式写清 `Why / Context / What`：

- Why：目标是什么，为什么现在做
- Context：已有上下文、约束、依赖、现状
- What：要交付什么，验收标准是什么

方向对齐后，再进入细节执行阶段。

如果当前意图仍然模糊，先做一轮短 discovery，而不是过早猜实现。

## 自动化决策分级

### 绿灯

- 低风险，不阻塞当前推进
- 可逆，或修复成本很低
- 不涉及工程审美拍板
- 不会污染多个下游模块

动作：

- 直接做
- 不打断人
- 只写入 review 面板 / 项目记录

### 黄灯

- 当前节点阻塞，但不紧急
- 还能先做别的安全工作
- 值得记录，但没到立刻打断的程度

动作：

- 不打断人
- 记录待决策项
- 继续推进其他安全工作
- 到 review 窗口统一处理

黄灯升级红灯的条件：

- 已经没有其他安全可推进工作
- 快到不可逆步骤
- 风险范围扩大
- 超过约定时限还没处理

### 红灯

- 阻塞且紧急
- 或虽不阻塞，但属于高风险异常
- 错误成本明显大于打断人的成本

触发条件：

- 不可逆操作 + 影响跨模块
- 两条路径都能做，但选哪个取决于人的工程审美判断
- 当前决策会污染多个下游模块
- 线上事故、数据污染、权限 / 安全问题
- 即将进入不可逆步骤，继续做会放大错误成本

动作：

- 立即 push 到企业 IM
- 在回复前不越过不可逆步骤
- 同时给出问题一句话、2-3 个选项、每个选项的优点 / 代价 / 风险、推荐方案、推荐理由、为什么现在必须拍板

红灯决策辅助偏好：

- 需要人拍板时，默认把不同决策的优缺点列出来，不只问“选哪个”。
- 推荐项要明确说清：为什么推荐、放弃其他项的代价是什么。

## Notion 职责分工

- 总览优先用多维表格，不用长文档总览快照
- Review page：只做轻量入口页，不承载长篇状态堆叠
- Execution doc page：稳定的执行文档和里程碑文档，供段落级评论交互
- Execution doc page 不需要额外“当前总览”顶栏，直接从最新执行同步记录开始
- Memory page：稳定协作偏好和工程审美的共享镜像
- 后续适合再加一层项目索引表：每个项目一行，每个 milestone 对应单独执行文档
- Notion 里的 review / execution / index 同步默认保留历史，走 append 机制，不做整页覆盖式重写
- Notion 历史同步默认倒序：最新进展在上，旧记录自然下沉
- 执行文档和 review 页面要保留固定导航区，但导航只保留“当前任务 / 核心进展 / 最近同步”这种一句话摘要，不把风险和 To Do 暴露成目录树
- 项目索引只在新 checkpoint 追加，不在同一任务 / 同一 checkpoint 上反复新增索引行
- 同一 checkpoint 内如果只是重复同步，默认跳过；只有任务、核心进展、风险状态或下一步发生了稳定变化，才算新的索引 checkpoint
- 同步时间默认用人的本地时间，不用裸 UTC / ISO 机器时间戳

不要把总览快照和具体段落评论混在同一张会被频繁整体重写的页面上。
Notion 是共享 review 面板，不是分叉的第二份真相。
- 正文承载内容本身，评论承载反馈和任务。
- 评论任务写在 Notion comment / discussion 里，不写进文档正文。
- 默认交互方式是对具体段落划词评论，再由 agent 轮询扫描评论并继续执行。
- 多 agent 协作时，不依赖 Notion `@` 某个 agent；评论分发靠路由契约。
- 评论路由优先级：评论前缀 `[agent: xxx]` > `@mention` 别名路由 > block/page 路由规则 > 默认 router agent。

## 工程审美

- 稳定优先
- 分层清晰
- 重评测
- 展示标签和检索标签分离
- 按真实产品目标做架构
- 渐进式披露
- 小额持续偿还
- 快速验证，修复便宜
- 机械约束大于反复调教
- 判断必须外化，写下来才算存在

## 本地部署偏好

- 本地 secrets 默认走 `.env.local / .env`，不额外引入单独 secrets 管理层。
- 本机如果是 macOS，本地常驻托管优先用 `launchd`；`systemd` 只适用于后续真的迁到 Linux 主机时再考虑。
- 红灯快速唤醒优先走本地系统通知，只要链路收益明显且无额外风险，就直接执行，不再被外部 IM 拓扑卡住。

## 产品决策风格

- 做正确的事，不做快的事
- 正确性和速度冲突时，选正确性
- 验收标准必须可验证，不只是可描述
- 好需求描述用户能做什么、系统如何表现，不写成实现步骤
- 大需求拆成可独立交付、可独立验证的最小单元
- 产品原则和红线必须显式写下来
- 定期检查产品漂移

## 结果呈现与注意力管理

- 一眼看不到重点的同步文档，默认算失败，先压缩再同步。
- 进展同步文档默认只保留三块：当前任务 / 当前进展，红黄绿灯决策状态，下一步计划。
- 文档内部尽量少用标题树；风险、To Do、已推进这类内容优先写成简短正文或 bullet，不把目录撑满。
- 更新按重要性和决策需求聚合，不按时间顺序刷屏。
- 能继续安全推进时，不发打断式状态 ping。
- 默认异步聚合，在固定 review 窗口集中消化。
- working software、working report、明确 diff，优先于空泛讨论。

## 沟通偏好

- 短句、直接，不用正式书面表达。
- 避免空泛、绕弯、咨询式语言。
- 多步逻辑拆成短句分批说，不要一次砸一大段。
- 代码太长时，先给骨架，再补细节。
- 表格只在真正必要时用，列表优先用 bullet。

## 更新 Checkpoints

以下情况后更新本文件：

- 关键架构或产品方向锁定后
- 完整纵切验收确认可复用模式后
- 评测或验收标准稳定变化后
- 协作偏好被明确表达或重复出现后
- 发现之前的假设是错的后

## 维护方式

- 当前本地文件 `docs/collaboration-memory.md` 是可 diff 的主版本。
- Notion memory page 是共享镜像，要和本地基线对齐。
- Obsidian collaboration memory note 是个人知识镜像，要和本地基线对齐。
- 新偏好优先修订已有原则，不堆重复 bullet。
- 只记录稳定、可复用、会影响未来 agent 行为的规则。
