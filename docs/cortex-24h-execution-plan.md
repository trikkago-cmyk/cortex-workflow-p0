# Cortex 未来 24 小时连续执行方案

最近更新：2026-05-09 15:12 CST

## 这份方案解决什么

这不是长期 roadmap。

这是一份接下来 `24 小时` 内可以直接执行的连续推进方案，目标只有一个：

`把 Cortex 当前 P0 从“方向与文档已基本收口”推进到“前台工作台主壳 + 关键状态模型 + 可验证联调路径”这三个层面同时收口。`

它要解决两个现实问题：

1. 当前产品定义已经越来越清楚，但前台工作台的核心界面和对象组织还没有正式落地。
2. 当前仓库能力很多，但容易在“继续补文档”和“真正往可用产品推进”之间来回摇摆。

所以这 24 小时的原则是：

- 不再继续发散新概念
- 不再重新定义主路径
- 只围绕已经确认的首页结构和 P0 闭环持续推进

---

## 当前已确认、不再反复讨论的前提

以下判断默认视为已拍板，在未来 24 小时内不再反复重开：

1. 首页只做 `一个工作台`，不是多个分散页面。
2. 首页默认是 `同一批 task 的两种视图`：
   - `按注意力`
   - `按线程`
3. `task` 是顶层管理单位，不用 `thread` 或 `run` 取代它。
4. `thread_key` 是机器稳定标识，`thread_label` 是人类可读标识。
5. 红灯才真正等待人工；黄灯不是同步待处理，而是 `绕行中 / 待回看`。
6. `run` 不做独立主看板，但它的执行脉搏必须在首页可见。
7. Cortex 主真相源仍然是 `本地 Markdown + SQLite`，Notion / Obsidian 继续只做镜像或协作入口。
8. 前台视觉必须避免“AI 模板味”，优先采用 `DESIGN.md` 式设计护栏，再进入页面实现。

## 24 小时目标

24 小时结束时，期望我们至少拿到下面 5 个结果：

1. 首页信息架构完全锁定
   - `顶部摘要卡`
   - `按注意力` 视图
   - `按线程` 视图
   - 任务卡字段模型
   - 线程分组头字段模型

2. 前台主壳进入可运行状态
   - `/workspace` 可以开始承载双视图骨架
   - 不再只是旧 `/dashboard` 的粗糙延续

3. 任务对象与线程归属模型对齐
   - `thread_key / thread_label`
   - task 与 thread 的归属关系
   - 首页分组和筛选规则

4. 执行状态和决策状态表达统一
   - `等待许可`
   - `系统处理中`
   - `已完成`
   - 卡片内显示 red / yellow / green、执行状态、最近 checkpoint、归属线程

5. 有一条清晰的可验证链路
   - 至少能验证首页聚合、任务分组、关键状态展示没有自相矛盾
   - 能跑通对应测试或最小手动验收步骤

## 执行原则

### 1. 默认持续执行，不中途停工确认

只要没有进入下面定义的红灯条件，就持续推进，不因为黄灯事项中断。

### 2. 文档和代码同步推进

不再出现“文档已经改了，但实现方向还是旧的”这种脱节。

### 3. 优先做产品主壳，不优先补角落能力

未来 24 小时内，优先级高于一切的事情是：

- 首页工作台主壳
- task / thread / run 信息表达
- 双视图切换

优先级低于这些的事情包括：

- 继续扩写解释型文档
- 继续发散新的 agent team 角色
- 继续做 Notion UI 接入层面的探索

### 3.1 前台视觉护栏

未来 24 小时内，`/workspace` 的前台实现默认遵守下面这些视觉约束：

1. 先定设计语言，再写页面
   - 参考 `DESIGN.md` 的方法，不直接拼现成 dashboard 模板
   - 先明确视觉气质、信息密度、色彩角色、层级关系，再做组件

2. 不做“AI 味很重”的默认 SaaS 看板
   - 不使用泛滥的紫蓝渐变 + 发光边框 + 圆角卡片堆叠
   - 不做过多悬浮阴影和“未来感控制台”套路
   - 不把内部工作台做成营销页

3. 风格目标是“内部执行中台”，不是“演示型产品海报”
   - 信息密度要高，但不混乱
   - 层级要强，但不靠夸张特效
   - 重点让红灯、停滞、在跑状态一眼可见

4. 优先参考成熟产品的结构感，而不是 AI 随机生成感
   - 参考 `DESIGN.md` 思路约束视觉语言
   - 参考你给的 `awesome-design-md` 资源，吸收其“先有风格文档，再让 agent 产出页面”的方法
   - 但不直接照抄某个品牌皮肤

5. 首页视觉关键词
   - 冷静
   - 克制
   - 执行导向
   - 文档工作流优先
   - 异常状态高可见

### 4. 非红灯默认自行裁定

如果遇到局部实现取舍，按下面规则处理：

- 不影响主路径：直接定
- 影响字段命名但不影响语义：直接定
- 影响首页主结构或 P0 范围：才升级成红灯

## 红黄绿执行规则

### 绿灯：直接做

包括但不限于：

- 首页字段细化
- 双视图文档继续落地
- 前端页面骨架搭建
- 接口字段补齐
- 测试补充
- README / docs 对齐

### 黄灯：记录后继续推进，不停

包括但不限于：

- 某个字段最终命名还可优化
- 卡片排序策略还有 2-3 种可能
- 首页视觉布局细节还未到最终版
- `按线程` 视图是列表分组还是折叠卡片组，仍可在实现中微调

处理方式：

- 先采用当前最稳方案实现
- 在文档或 TODO 中记录待回看点
- 不阻断主链路

### 红灯：必须停下来同步

只有下面几类情况才算红灯：

1. 需要推翻当前首页双视图架构
2. 需要把 `task` 改成别的顶层主单位
3. 需要大范围重写现有数据模型，导致过去文档全部失效
4. 发现真实代码结构与当前 PRD 假设完全冲突，无法在 24 小时内通过局部改造解决
5. 要做有破坏性的清理、删除或迁移，会直接影响现有可运行能力

只要没遇到这些，就默认不停。

## Phase 1：首页规格定稿（第 1 - 4 小时）

目标：

- 把前台首页的产品规格收口到可以直接编码的粒度

要完成的内容：

1. 顶部摘要卡定稿
   - 活跃项目 / 线程
   - 总任务数
   - 系统处理中
   - 等待许可
   - 疑似停滞
   - 已完成

2. `按注意力` 视图定稿
   - `等我拍板`
   - `系统处理中`
   - `已完成`

3. `按线程` 视图定稿
   - 线程分组头字段
   - 线程内 task 排序
   - 分组展开逻辑

4. 任务卡字段模型定稿
   - task title
   - decision signal
   - execution status
   - thread_label
   - owner agent（可空）
   - 最近 checkpoint / 最近更新时间
   - 下一步动作

5. 可空字段策略定稿
   - 不强求所有 task 都有当前 agent
   - 不强求所有 task 都有 workflow 节点
   - 缺失时展示为 `未分配 / 待路由 / 未启动`

产出：

- 规格文档补齐
- 首页字段清单
- 页面状态映射表

验收：

- 不需要再讨论“首页到底展示什么”

## Phase 2：数据与接口对齐（第 4 - 8 小时）

目标：

- 把首页双视图需要的数据结构明确化

要完成的内容：

4. 明确线程分组规则
   - 同 `thread_key` 聚合
   - `thread_label` 只做人类展示

5. 对齐 server / task-dashboard / store 所需改造点

产出：

- 接口字段定义
- 聚合规则清单
- 数据模型与页面字段映射关系

验收：

- 首页双视图的数据来源明确，不再靠临时拼装猜字段
